# 01. 计算机网络：TCP、UDP、HTTP、HTTPS

## 1. TCP 和 UDP 区别？

### 标准回答
- TCP 面向连接、可靠传输、字节流
- UDP 无连接、不保证可靠、面向报文、开销更小

### 对比维度
- 连接管理：TCP 有三次握手四次挥手，UDP 无
- 可靠性：TCP 有重传、确认、排序、流控、拥塞控制
- 性能：UDP 头部小、时延低，但可靠性需要业务层补

## 2. 为什么 TCP 是可靠的？

### 标准回答
TCP 通过以下机制实现可靠传输：
- 序列号
- 确认应答 ACK
- 超时重传
- 校验和
- 乱序重组
- 流量控制
- 拥塞控制

## 3. TCP 为什么三次握手，不是两次？

### 标准回答
三次握手的核心目的是让通信双方都确认：
1. 自己的发送能力正常
2. 自己的接收能力正常
3. 对方的发送能力正常
4. 对方的接收能力正常

### 为什么两次不够？
两次无法让服务端确认客户端已经收到自己的确认，也就无法可靠建立双向通信状态。

## 4. TCP 为什么四次挥手？

### 标准回答
TCP 是全双工连接，双方关闭发送通道通常是独立过程，所以需要四次挥手。

### 过程
1. A 发送 FIN
2. B 回 ACK
3. B 处理完剩余数据后发送 FIN
4. A 回 ACK

## 5. TIME_WAIT 为什么要存在？

### 标准回答
TIME_WAIT 的主要作用：
- 保证最后一个 ACK 能让被动关闭方收到
- 防止旧连接中的延迟报文影响后续新连接

## 6. 为什么会有粘包和拆包？

### 标准回答
因为 TCP 是字节流协议，不保留应用层消息边界。发送方多次写入和接收方读取粒度不一致，就会出现粘包/拆包。

### 解决方式
- 固定长度协议
- 分隔符协议
- 消息头 + 消息体长度字段

## 7. 流量控制和拥塞控制区别？

### 流量控制
点对点，解决“接收方来不及处理”的问题，靠滑动窗口。

### 拥塞控制
面向网络整体，解决“网络链路过载”的问题，典型算法有慢启动、拥塞避免等。

## 8. 什么是滑动窗口？

### 标准回答
滑动窗口用于提高传输效率，允许发送方在未收到每个分组确认前连续发送多个分组。

### 面试价值
它同时关联：
- 流量控制
- 累积确认
- 吞吐提升

## 9. HTTP 和 HTTPS 区别？

### 标准回答
HTTPS = HTTP + TLS/SSL。
HTTPS 在应用层协议外增加加密与身份认证，主要提供：
- 机密性
- 完整性
- 身份认证

## 10. HTTPS 建连大致过程？

### 标准回答
简化理解：
1. 客户端发起 TLS 握手
2. 服务端返回证书和公钥相关信息
3. 客户端验证证书
4. 协商对称密钥
5. 后续数据用对称加密传输

### 面试注意
- 真正细节会因 TLS 版本不同而不同
- 不要把“公钥加密所有传输数据”当成标准答案，实际大流量数据通常用对称加密

## 11. HTTP 常见状态码有哪些？

### 高频
- 200 OK
- 301/302 重定向
- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden
- 404 Not Found
- 500 Internal Server Error
- 502 Bad Gateway
- 503 Service Unavailable
- 504 Gateway Timeout

## 12. GET 和 POST 区别？

### 正确回答
语义上：
- GET 用于获取资源
- POST 用于提交资源

### 易错点
- 不是“GET 一定没有请求体，POST 一定安全”这种说法
- 核心是语义、幂等性、缓存行为与实践约定
