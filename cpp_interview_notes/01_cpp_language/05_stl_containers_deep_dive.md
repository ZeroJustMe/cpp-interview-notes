# 05. STL 容器深挖：vector / deque / list / map / unordered_map

## 1. vector 的底层实现是什么？

### 标准回答
`vector` 底层通常是连续内存数组，维护 `begin`、`end`、`cap` 等信息，支持随机访问，尾部插入均摊 O(1)。

### 追问点
- 为什么随机访问快：连续内存，可按偏移直接定位
- 为什么尾插均摊 O(1)：扩容不是每次发生，而是按增长策略批量扩
- 为什么中间插入慢：需要整体搬移元素

## 2. vector 扩容时发生了什么？

- 申请更大的一段连续内存
- 把旧元素拷贝/移动到新空间
- 释放旧空间
- 更新内部指针

### 面试加分点
如果元素支持 noexcept move，很多实现会优先移动而不是拷贝，以降低代价。

## 3. deque 和 vector 区别？

### vector
- 单块连续内存
- 随机访问快
- 头部插入删除差

### deque
- 分段连续存储 + 中控结构
- 头尾插入删除都更友好
- 不保证整体物理连续

### 适用场景
- 需要频繁头尾插入：`deque`
- 需要最大化连续内存收益：`vector`

## 4. list 为什么现代工程里用得少？

### 原因
- 每个节点额外前驱/后继指针，内存占用大
- 节点离散，cache locality 差
- 遍历性能往往明显不如 `vector`

### 高分表达
`list` 的理论复杂度好看，但现代 CPU 更看重局部性；很多实际业务中，`vector` 即使有搬移成本，整体仍更快。

## 5. set / map 为什么通常基于红黑树？

### 标准回答
因为红黑树在插入、删除、查找上能提供稳定的 O(logN) 性能，并保持元素有序。

## 6. unordered_map 为什么不一定比 map 快？

### 原因
- 哈希冲突会退化
- rehash 有额外成本
- 不支持顺序遍历语义
- 对 key 的哈希质量敏感
- 某些数据规模/访问模式下 cache 行为未必更优

## 7. 什么场景优先 map？

- 需要有序遍历
- 需要范围查找 / lower_bound / upper_bound
- 需要更稳定的最坏复杂度

## 8. 什么是 iterator invalidation？

### 常见考点
- `vector` 扩容：原有迭代器/引用/指针可能全部失效
- `deque` 中间插删：很多迭代器失效
- `list`：除删除点自身外，其它节点通常稳定
- `unordered_map` rehash：迭代器常失效

## 9. 面试提示

容器题不要只背复杂度表，一定补上：
- 连续内存 vs 离散内存
- cache locality
- 迭代器稳定性
- 扩容 / rehash / 节点分配成本
