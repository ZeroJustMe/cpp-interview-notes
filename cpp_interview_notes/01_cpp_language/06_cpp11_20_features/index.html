    06. C++11 ~ C++20 高频新特性（深挖版）¶

    新特性题最容易掉进一个坑：把 C++11 到 C++20 背成版本功能清单。

    但面试官真正想听的是：

    • 这些特性为什么会出现
    • 它们解决了哪些旧问题
    • 它们带来了什么新的边界和代价

    所以这一章不追求"面面俱到地背标准更新"，而是聚焦面试最常问、工程里最常用、最能体现现代 C++ 思维变化的那批特性。

    本章按三层难度组织：

    • ⭐ 基础必会 — C++11 常用特性、nullptr、auto、lambda、emplace_back
    • 🔥 高频追问 — constexpr、optional/variant/any、结构化绑定、string_view
    • 💎 深度难点 — C++20 concepts、模块化、Ranges

    ⭐ 第一层：基础必会¶

    1. C++11 你常用哪些特性？¶

    高频回答¶

    • auto
    • nullptr
    • 范围 for
    • 右值引用 / 移动语义
    • lambda
    • std::thread / mutex / atomic
    • 智能指针
    • constexpr
    • override / final
    • 列表初始化

    面试更强的答法¶

    不要只背名单，最好按"解决了什么问题"分类：

    • 减少类型噪音：auto
    • 提高接口安全：override、nullptr
    • 改善资源与性能：移动语义、智能指针
    • 提供现代并发库：thread / mutex / atomic
    • 提升表达力：lambda、范围 for

    2. nullptr 和 NULL 的区别？¶

    标准回答¶

    nullptr 是专门的空指针字面量，类型安全，能避免和整数重载混淆；NULL 往往只是 0 或 ((void*)0) 的宏。

    为什么这题重要？¶

    因为它体现了现代 C++ 的一个方向：

    用类型系统替代历史包袱和模糊语义。

    经典场景¶

    函数重载时，NULL 可能被当成整数参与匹配，而 nullptr 明确表达"空指针语义"。

    3. auto 会不会降低代码可读性？¶

    合理回答¶

    取决于场景。

    • 类型显而易见时，auto 可以减少噪音
    • 类型不明显时，滥用会降低可读性

    为什么 auto 会成为高频问题？¶

    因为它不是单纯省字数，而是在考你对"显式类型 vs 表达式驱动类型推导"的态度。

    更成熟的表达¶

    auto 的价值在于让代码聚焦在真正重要的语义上，但前提是推导结果对读者来说不应成为额外负担。

    4. lambda 的本质是什么？¶

    标准回答¶

    lambda 本质上会生成一个匿名函数对象（闭包类型），捕获变量会成为该对象的成员。

    为什么它改变了现代 C++ 写法？¶

    因为它极大降低了"写一个临时小策略对象"的成本，使得：

    • STL 算法
    • 回调
    • 异步任务
    • 局部策略封装

    变得更自然。

    追问点¶

    • 值捕获是拷贝
    • 引用捕获要注意生命周期
    • mutable 影响值捕获副本可修改性

    高分点¶

    lambda 不是"匿名函数语法糖"这么简单，它本质上是更轻量地生成函数对象，和泛型算法、回调体系天然契合。

    5. emplace_back 和 push_back 区别？¶

    标准回答¶

    push_back 传入一个对象；emplace_back 直接在容器尾部原地构造对象，可能减少一次临时对象构造/移动。

    易错点¶

    不是所有场景 emplace_back 都更快，具体还要看参数和构造路径。

    更成熟的回答¶

    emplace_back 的优势在于减少中间对象和构造路径冗余，但如果你本来就已经有一个现成对象，push_back(std::move(x)) 也可能同样合理。

    🔥 第二层：高频追问¶

    6. constexpr 是什么？¶

    标准回答¶

    constexpr 表示表达式、变量或函数在满足条件时可用于编译期求值。

    为什么它重要？¶

    因为它反映了现代 C++ 的另一个方向：

    • 尽量把可确定的逻辑前移到编译期
    • 提高性能
    • 增强类型与接口表达力

    但不要说太死¶

    constexpr 不代表"永远在编译期执行"，更准确是：

    • 在满足条件时允许编译期求值
    • 否则也可能在运行期执行

    7. optional / variant / any 分别解决什么问题？¶

    optional¶

    表达"有值/无值"语义，比魔法值更清晰。

    variant¶

    类型安全的联合体，表示"多种候选类型之一"。

    any¶

    可装任意类型，但类型擦除带来运行时成本。

    为什么这三个常一起问？¶

    因为它们都在解决"值的表达方式"问题，但类型约束强弱不同：

    • optional：一个类型 + 空状态
    • variant：若干已知候选类型
    • any：任意类型，最灵活也最弱约束

    8. C++17 结构化绑定有什么用？¶

    例子¶

    auto [x, y] = p;

    价值¶

    有利于简化 pair / tuple 的解包，提高可读性。

    为什么这不只是语法糖？¶

    因为它让很多 STL 场景的表达更自然：

    • 遍历 map
    • 处理多返回值
    • 解构中间结果

    这会直接改善代码可读性和意图表达。

    9. string_view 的优缺点？¶

    优点¶

    • 不拷贝字符串
    • 只是一段视图，传参成本低

    风险¶

    • 不拥有底层数据
    • 原字符串失效后，string_view 也悬空

    为什么这题很爱考？¶

    因为它正好体现现代 C++ 里"零拷贝 / 轻量视图"与"生命周期风险"是一体两面的。

    高分点¶

    string_view 的价值是避免不必要拷贝，但它本质是观察者，不拥有底层字符串，所以生命周期永远是第一注意点。

    💎 第三层：深度难点¶

    10. C++20 concepts 的意义是什么？¶

    标准回答¶

    concepts 用于约束模板参数，改善模板错误信息，并让泛型接口语义更清晰。

    为什么它比传统 SFINAE 更重要？¶

    因为过去模板约束常常写得：

    • 隐晦
    • 错误信息难看
    • 接口意图不直观

    而 concepts 更接近"把模板前提条件写成语言级接口契约"。

    一句总结¶

    concepts 的价值不只是报错更友好，而是让泛型代码终于能像普通接口那样显式表达前提条件。

    11. 一组典型追问链¶

    1. C++11 你最常用哪些特性？
    2. nullptr 为什么比 NULL 更安全？
    3. auto 为什么有人喜欢、有人反感？
    4. lambda 为什么改变了现代 C++ 风格？
    5. emplace_back 为什么不一定总更快？
    6. constexpr 是"编译期常量"还是"可编译期求值"？
    7. optional / variant / any 怎么选？
    8. string_view 为什么既高效又危险？
    9. concepts 为什么比 SFINAE 更好？

    12. 一份更像面试现场的总结回答¶

    C++11 到 C++20 的核心变化，不只是"多了很多新语法"，而是语言整体在往更强类型表达、更低资源开销、更现代泛型和并发模型上演进。nullptr、override 代表接口安全，移动语义和 emplace 代表性能意识，lambda 和结构化绑定代表表达力提升，optional/variant/string_view 代表更细腻的数据语义，而 concepts 则代表模板约束终于从技巧走向显式语言机制。真正好的回答，不是背版本清单，而是能说清这些特性分别在修补什么旧问题。

    13. 复习建议¶

    至少做到：

    • 不只背新特性名单，而是能说出它解决了什么旧问题
    • 能把 nullptr、lambda、移动语义、string_view 讲到工程边界
    • 能区分 optional / variant / any 的语义层次
    • 能把 concepts 说成泛型接口约束升级，而不是"更好看的 SFINAE"

    做到这里，新特性题就不会再像版本背诵题，而会更像现代 C++ 理解题。
