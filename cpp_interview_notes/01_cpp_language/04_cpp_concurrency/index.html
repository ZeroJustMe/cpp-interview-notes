    04. C++ 并发编程：线程、锁、原子、内存模型（深挖版）¶

    C++ 并发是最容易"会写一点线程代码"和"真正理解并发语义"之间拉开差距的地方。很多人会背：

    • mutex 保证线程安全
    • atomic 更轻量
    • 条件变量要防虚假唤醒
    • memory_order 有几种

    但真正的难点在于：

    • 线程安全到底怎么定义？
    • race condition 和 data race 差在哪？
    • 为什么 atomic 不等于整体线程安全？
    • happens-before 到底在保证什么？
    • 为什么无锁编程不一定更快？

    这一章的重点，就是把"并发 API"提升成"并发语义理解"。

    本章按三层难度组织：

    • ⭐ 基础必会 — 线程安全、竞态条件、mutex、死锁
    • 🔥 高频追问 — 条件变量、读写锁、原子操作、lock_guard/unique_lock
    • 💎 深度难点 — 内存序、happens-before、无锁编程

    ⭐ 第一层：基础必会¶

    1. 线程安全怎么定义？¶

    标准回答¶

    线程安全指多个线程并发访问某段代码或某个对象时，不会因竞态条件导致数据错误、崩溃或未定义行为，并且结果符合预期同步语义。

    更成熟的理解¶

    线程安全不是一句"加了锁就行"，它至少包含两层：

    • 正确性：不会读到脏状态、不会破坏不变式、不会 UB
    • 语义性：结果要符合设计者想要的并发语义

    面试高分点¶

    线程安全不是"程序没崩"就算数，而是并发访问下仍然保持正确状态和预期可见性。

    2. 什么是竞态条件（race condition）？¶

    标准回答¶

    竞态条件指程序结果依赖多个线程执行时序，而这种时序不可控，导致结果不稳定。

    与 data race 区别¶

    • race condition 是更广义概念
    • data race 是 C++ 内存模型里的特定术语：多个线程并发访问同一内存位置，至少一个写，且缺乏同步关系，结果是未定义行为

    为什么这个区分重要？¶

    因为很多逻辑竞态：

    • 即使没有触发 data race
    • 也可能业务结果错误

    反过来，data race 是语言层面更硬的红线，一旦发生就是 UB，不能拿"我测试没出问题"安慰自己。

    3. mutex 和 atomic 的区别？¶

    标准回答¶

    • mutex 适合保护临界区和复杂共享状态
    • atomic 适合对单个原子变量进行无锁原子操作

    为什么原子变量不等于整个逻辑线程安全？¶

    如果多个共享变量之间存在一致性约束，仅把一个变量做成 atomic 往往不够。

    比如：

    • 一个变量表示数量
    • 另一个变量表示状态

    两者需要同步演化，单独原子更新一个并不能保证整体逻辑正确。

    一句总结¶

    atomic 解决的是单个原子对象的并发访问问题，mutex 更适合保护跨多个状态的不变式。

    4. 什么是死锁？如何避免？¶

    标准回答¶

    死锁是多个线程互相等待对方释放资源，导致永久阻塞。

    四个必要条件¶

    1. 互斥
    2. 占有且等待
    3. 不可剥夺
    4. 环路等待

    避免方法¶

    • 固定加锁顺序
    • 减少锁粒度和持锁时间
    • 使用 std::lock / scoped_lock
    • 避免嵌套锁设计

    面试高分点¶

    死锁不是"某个 API 用错了"这么简单，而是资源获取顺序设计出了环路。解决它通常靠设计约束，而不是运行时碰运气。

    5. 什么是条件变量？¶

    标准回答¶

    条件变量用于线程间等待/通知机制，让线程在某个条件未满足时休眠，满足时被唤醒。

    标准用法¶

    必须搭配互斥锁和谓词条件使用，防止虚假唤醒。

    cv.wait(lock, [] { return ready; });

    它解决的不是"互斥"，而是"等待条件变化"¶

    这是很多人容易混淆的点：

    • mutex 解决互斥进入
    • condition_variable 解决高效等待某个状态变成真

    如果没有条件变量，线程只能：

    • 忙等
    • 反复轮询

    这会浪费 CPU。

    🔥 第二层：高频追问¶

    6. 什么是虚假唤醒？¶

    标准回答¶

    线程在没有真正收到有效通知或条件仍未满足时从等待状态返回，称为虚假唤醒。

    正确写法¶

    永远用 while 或带谓词版本的 wait 检查条件，而不是假设唤醒即条件成立。

    为什么这题老被问？¶

    因为它能测出你是：

    • 真的写过并发代码
    • 还是只记得 API 名字

    一句总结¶

    条件变量的正确用法，不是"等到被叫醒"，而是"被叫醒后重新确认条件是否真的成立"。

    7. 什么是原子操作？¶

    标准回答¶

    原子操作指操作不可被其他线程在中间观察到"半完成状态"，要么完全发生，要么完全没发生。

    但原子性只解决一部分问题¶

    原子操作保证的是：

    • 单个操作不会被撕裂观察

    它不自动保证：

    • 多步逻辑整体正确
    • 多变量之间关系一致
    • 更高层业务约束成立

    高分点¶

    原子性解决的是"操作不可分割"，不是"整个并发逻辑自动正确"。

    8. memory_order 了解多少？¶

    最少要答出的几个¶

    • memory_order_relaxed：只保证原子性，不保证额外顺序
    • memory_order_acquire：获取语义，保证后续读写不被重排到前面
    • memory_order_release：释放语义，保证前面的写不被重排到后面
    • memory_order_acq_rel：读改写操作常用
    • memory_order_seq_cst：最强、最直观的全序一致性语义

    如果没写过 lock-free，要怎么答更稳？¶

    不要乱吹底层指令细节。更稳的讲法是：

    • relaxed 只有原子性
    • acquire/release 用于建立线程间同步边
    • seq_cst 最容易理解，但可能限制优化更多

    高分点¶

    memory order 的关键，不是背名字，而是理解"原子性"和"可见性/顺序性"是两回事。

    💎 第三层：深度难点¶

    9. 什么是 happens-before？¶

    标准回答¶

    happens-before 描述两个操作之间的先后可见性关系。如果 A happens-before B，那么 B 必须能看到 A 的结果，且编译器/CPU 不能把这层关系打乱。

    常见建立方式¶

    • 同线程程序顺序
    • 锁的解锁与加锁
    • 原子变量 release/acquire 同步
    • 线程创建与 join

    为什么这是并发核心概念？¶

    因为并发真正难的不是"线程同时跑"，而是：

    • 哪些写对别的线程可见
    • 哪些顺序是必须保持的
    • 哪些重排是被允许的

    happens-before 正是在回答这件事。

    10. 线程池为什么常考？¶

    标准回答¶

    线程池的核心价值是复用线程、减少频繁创建销毁开销、控制并发度、统一调度任务。

    面试常追问¶

    • 任务队列如何设计
    • 如何优雅关闭线程池
    • 阻塞队列怎么做
    • 拒绝策略怎么设计
    • worker 线程异常如何处理

    为什么它是并发综合题？¶

    因为线程池把很多问题揉在一起了：

    • 同步
    • 调度
    • 生命周期
    • 背压
    • 资源控制

    所以它几乎是 C++ 并发面试的经典大题。

    11. 读写锁适合什么场景？¶

    标准回答¶

    读多写少场景适合读写锁，让多个读线程并发访问，而写线程独占。

    但不要说成"读多写少就一定上"¶

    因为读写锁不是总比互斥锁好：

    • 实现更复杂
    • 写饥饿问题要考虑
    • 读锁本身也有开销
    • 在某些负载下未必赢普通 mutex

    一句总结¶

    读写锁是特定访问模式下的优化工具，不是更高级的万能锁。

    12. 无锁编程是不是一定更快？¶

    标准回答¶

    不是。无锁编程虽然减少阻塞，但会引入 CAS 重试、缓存一致性流量、ABA 问题和更复杂的正确性验证。很多实际业务里，设计良好的锁反而更稳定、更易维护。

    为什么很多人会误会？¶

    因为"无锁"听起来像：

    • 没有阻塞
    • 没有上下文切换
    • 所以一定快

    但真实世界里还要看：

    • CAS 失败重试成本
    • cache line 抖动
    • 热点竞争
    • 复杂性和可维护性

    高分点¶

    无锁不等于无代价；它只是把"等待锁"的成本，换成了"重试、缓存流量和正确性复杂度"的成本。

    5. happens-before 到底在保证什么？

    14. 一份更像面试现场的总结回答¶

    C++ 并发真正难的，不是 API 数量多，而是语义层复杂：你既要保证没有 data race，又要保证更高层业务状态的一致性。mutex 擅长保护复杂共享状态，atomic 擅长表达单点原子同步，但不能替代整体并发设计；条件变量解决的是等待条件变化，happens-before 则定义了跨线程可见性和顺序边界。真正成熟的回答，不是说"我会用锁和原子"，而是能说清什么时候该用、为什么这样用、这样做的代价是什么。

    15. 复习建议¶

    至少做到：

    • 能稳定区分 race condition 和 data race
    • 能解释 atomic 为什么不是并发万金油
    • 能用 happens-before 解释线程间可见性
    • 能把条件变量和虚假唤醒讲明白
    • 能说明无锁编程的收益和代价

    做到这里，这一章就不再只是线程 API 记忆题，而开始接近真正的并发理解。
