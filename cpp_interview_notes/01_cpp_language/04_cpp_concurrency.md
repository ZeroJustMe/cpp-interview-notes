# 04. C++ 并发编程：线程、锁、原子、内存模型

## 1. 线程安全怎么定义？

### 标准回答
线程安全指多个线程并发访问某段代码或某个对象时，不会因竞态条件导致数据错误、崩溃或未定义行为，并且结果符合预期同步语义。

## 2. 什么是竞态条件（race condition）？

### 标准回答
竞态条件指程序结果依赖多个线程执行时序，而这种时序不可控，导致结果不稳定。

### 与 data race 区别
- race condition 是更广义概念
- data race 是 C++ 内存模型里的特定术语：多个线程并发访问同一内存位置，至少一个写，且缺乏同步关系，结果是未定义行为

## 3. mutex 和 atomic 的区别？

### 标准回答
- `mutex` 适合保护临界区和复杂共享状态
- `atomic` 适合对单个原子变量进行无锁原子操作

### 易错点
原子变量不等于整个逻辑线程安全。
如果多个共享变量之间存在一致性约束，仅把一个变量做成 atomic 往往不够。

## 4. 什么是死锁？如何避免？

### 标准回答
死锁是多个线程互相等待对方释放资源，导致永久阻塞。

### 四个必要条件
1. 互斥
2. 占有且等待
3. 不可剥夺
4. 环路等待

### 避免方法
- 固定加锁顺序
- 减少锁粒度和持锁时间
- 使用 `std::lock` / `scoped_lock`
- 避免嵌套锁设计

## 5. 什么是条件变量？

### 标准回答
条件变量用于线程间等待/通知机制，让线程在某个条件未满足时休眠，满足时被唤醒。

### 标准用法
必须搭配互斥锁和谓词条件使用，防止虚假唤醒。

```cpp
cv.wait(lock, [] { return ready; });
```

## 6. 什么是虚假唤醒？

### 标准回答
线程在没有真正收到有效通知或条件仍未满足时从等待状态返回，称为虚假唤醒。

### 正确写法
永远用 while 或带谓词版本的 wait 检查条件，而不是假设唤醒即条件成立。

## 7. 什么是原子操作？

### 标准回答
原子操作指操作不可被其他线程在中间观察到“半完成状态”，要么完全发生，要么完全没发生。

## 8. memory_order 了解多少？

### 最少要答出的几个
- `memory_order_relaxed`：只保证原子性，不保证额外顺序
- `memory_order_acquire`：获取语义，保证后续读写不被重排到前面
- `memory_order_release`：释放语义，保证前面的写不被重排到后面
- `memory_order_acq_rel`：读改写操作常用
- `memory_order_seq_cst`：最强、最直观的全序一致性语义

### 面试建议
如果没有真实 lock-free 经验，不要硬讲过深实现细节；讲清 acquire-release 的作用边界更重要。

## 9. 什么是 happens-before？

### 标准回答
happens-before 描述两个操作之间的先后可见性关系。如果 A happens-before B，那么 B 必须能看到 A 的结果，且编译器/CPU 不能把这层关系打乱。

### 常见建立方式
- 同线程程序顺序
- 锁的解锁与加锁
- 原子变量 release/acquire 同步
- 线程创建与 join

## 10. 线程池为什么常考？

### 标准回答
线程池的核心价值是复用线程、减少频繁创建销毁开销、控制并发度、统一调度任务。

### 面试常追问
- 任务队列如何设计
- 如何优雅关闭线程池
- 阻塞队列怎么做
- 拒绝策略怎么设计
- worker 线程异常如何处理

## 11. 读写锁适合什么场景？

### 标准回答
读多写少场景适合读写锁，让多个读线程并发访问，而写线程独占。

### 注意
读写锁不是总比互斥锁好：
- 实现更复杂
- 写饥饿问题要考虑
- 读锁本身也有开销

## 12. 无锁编程是不是一定更快？

### 标准回答
不是。无锁编程虽然减少阻塞，但会引入 CAS 重试、缓存一致性流量、ABA 问题和更复杂的正确性验证。很多实际业务里，设计良好的锁反而更稳定、更易维护。
