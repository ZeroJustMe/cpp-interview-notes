# 02. 值类别、引用、移动语义、完美转发、智能指针

## 1. 指针和引用有什么区别？

### 标准回答
- 指针本质上保存地址，可为空，可修改指向
- 引用是对象的别名，定义时必须初始化，通常不能改绑
- 语法上引用更像对象本身，指针需要显式解引用

### 深挖细节
1. 引用不一定在底层等价于指针，这取决于编译器实现
2. 引用更像一种语言级约束，常用于表达“不能为空、必须绑定”的语义
3. 接口设计中：
   - 需要可空语义：用指针
   - 需要必有对象语义：优先引用

## 2. 什么是左值、右值、将亡值？

### 标准回答
- 左值：有持久身份、可以取地址的表达式
- 右值：通常是临时值或纯值
- 将亡值（xvalue）：即将被移动的值

### 更准确的理解
现代 C++ 值类别可分为：
- glvalue
- prvalue
- xvalue

但面试里通常重点考：
- 左值引用绑定左值
- 右值引用绑定右值
- 为什么引入右值引用

## 3. 为什么需要右值引用？

### 标准回答
右值引用的核心目标是支持移动语义和完美转发，避免临时对象的深拷贝，提升性能。

### 典型场景
```cpp
std::vector<std::string> v;
std::string s = "hello";
v.push_back(s);            // 拷贝
v.push_back(std::move(s)); // 移动
```

## 4. std::move 到底做了什么？

### 标准回答
`std::move` 本质上只是一个类型转换，把表达式转换成右值语义，从而让编译器更倾向调用移动构造或移动赋值。它本身不移动任何资源。

### 易错点
- 不要说“`std::move` 会把对象搬走”
- 真正搬资源的是目标类型的移动构造函数/移动赋值运算符

## 5. 被 move 之后的对象还能用吗？

### 标准回答
能用，但通常只保证处于“有效但值未指定”状态。

### 正确理解
- 可以析构
- 可以重新赋值
- 但不应该依赖其原有业务值

## 6. 拷贝构造和移动构造的区别？

### 标准回答
- 拷贝构造：复制资源
- 移动构造：转移资源所有权，避免深拷贝

### 例子
对持有堆内存的类：
- 拷贝构造要新申请内存并复制内容
- 移动构造可直接接管指针，再把源对象置空

## 7. 什么是完美转发？

### 标准回答
完美转发指模板函数在转发参数时保留参数原本的值类别（左值/右值）和 cv 限定信息。

### 典型写法
```cpp
template <typename T>
void wrapper(T&& arg) {
    foo(std::forward<T>(arg));
}
```

### 核心点
- `T&&` 在模板推导上下文中可能是转发引用
- `std::forward<T>` 根据 `T` 的推导结果决定转发成左值还是右值

## 8. 什么是引用折叠？

### 规则总结
- `T& &` -> `T&`
- `T& &&` -> `T&`
- `T&& &` -> `T&`
- `T&& &&` -> `T&&`

一句话：**只要有左值引用参与，最终结果通常折叠成左值引用。**

## 9. RAII 是什么？

### 标准回答
RAII（Resource Acquisition Is Initialization）是 C++ 的核心资源管理思想：把资源获取与对象生命周期绑定，在构造时获取，在析构时释放。

### 资源不限于内存
- 文件句柄
- socket
- 锁
- 数据库连接
- 事务

### 面试高分点
RAII 的本质不是“智能指针”，而是“把资源管理纳入类型系统和作用域语义”。

## 10. unique_ptr / shared_ptr / weak_ptr 区别？

### unique_ptr
- 独占所有权
- 不能拷贝，可移动
- 开销小，生命周期清晰

### shared_ptr
- 共享所有权
- 通过引用计数控制生命周期
- 更灵活，但有额外成本

### weak_ptr
- 弱引用，不增加引用计数
- 用于观察对象、打破循环引用

## 11. 为什么项目里不要滥用 shared_ptr？

### 标准回答
因为 `shared_ptr` 虽然方便，但会带来：
- 控制块和引用计数开销
- 生命周期边界模糊
- 循环引用风险
- 在并发环境下可能出现额外原子计数成本

### 更好的表达
默认优先使用 `unique_ptr` 表达唯一所有权，只有在确实需要共享生命周期时再考虑 `shared_ptr`。

## 12. shared_ptr 的循环引用怎么产生？

### 例子
A 持有 B 的 `shared_ptr`，B 持有 A 的 `shared_ptr`。

### 后果
双方引用计数都不为 0，即使外部已不再使用，也不会释放。

### 解决
- 一边改为 `weak_ptr`
- 更根本地重构所有权关系

## 13. make_shared 为什么常被推荐？

### 标准回答
`make_shared` 一般更高效、更安全，原因包括：
- 往往只需要一次内存分配
- 异常安全更好
- 代码更简洁

### 注意
这更多是常见实现优化，不是“语言标准强制只有一次分配”。

## 14. 复习提示

这一章是真正的 C++ 深水区，高频且容易被追问：
- `std::move` 只是类型转换
- 完美转发依赖转发引用 + `std::forward`
- `shared_ptr` 不是银弹
- RAII 才是资源管理的根
