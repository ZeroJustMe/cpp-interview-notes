# 03. STL、模板、编译链接、const/static/inline

## 1. vector 和 list 怎么选？

### 标准回答
工程上通常优先考虑 `vector`，因为连续内存带来更好的 cache locality 和更高遍历性能。`list` 虽然在已知位置插删为 O(1)，但节点离散、额外指针开销大，很多场景下整体性能不如 `vector`。

### 面试不要机械背
不要直接说“list 插入删除一定更快”。
要补一句：
- 还要看插删位置是否已知
- 看遍历频率
- 看 cache 命中
- 看元素大小和移动成本

## 2. vector 为什么扩容要按倍数增长？

### 标准回答
倍数增长是为了在空间和时间之间折中，使 `push_back` 的均摊复杂度接近 O(1)。

### 深挖细节
- 如果每次只扩 1 个，频繁重新分配和拷贝，代价过高
- 倍数增长能显著减少扩容次数
- 不同实现的增长因子不一定完全一致

## 3. reserve 和 resize 区别？

### reserve
- 只预留容量，不改变元素个数
- 不会默认构造新元素

### resize
- 改变容器大小
- 可能新增元素并进行默认构造

## 4. map 和 unordered_map 区别？

### 标准回答
- `map` 通常基于红黑树实现，有序，复杂度稳定 O(logN)
- `unordered_map` 通常基于哈希表实现，平均 O(1)，但最坏可能退化

### 深挖细节
选择时要看：
- 是否要求有序
- 是否需要范围查询
- 是否关心最坏复杂度
- 哈希冲突和 rehash 成本是否可接受

## 5. 什么是迭代器失效？

### 标准回答
迭代器失效是指容器修改后，原有迭代器/引用/指针不再指向有效元素或不再安全使用。

### 常考点
- `vector` 扩容后，原有迭代器、引用、指针可能全部失效
- `erase` 后，被删除位置及后续元素迭代器常失效
- `list` 插入删除通常不会让其他节点迭代器失效
- `unordered_map` rehash 后大量迭代器失效

## 6. 模板为什么一般写在头文件里？

### 标准回答
因为模板实例化发生在使用点，编译器在看到具体类型参数时，需要模板完整定义来生成代码。所以模板通常放在头文件中。

## 7. 模板和宏的区别？

### 标准回答
- 宏是预处理阶段的文本替换，没有类型检查
- 模板是编译期泛型机制，受类型系统约束，有作用域和重载决议

### 面试高分点
模板体现的是“类型安全的编译期抽象”，而不是简单替换。

## 8. 什么是模板特化和偏特化？

### 标准回答
- 全特化：为某个具体类型提供专门实现
- 偏特化：为一类模板参数模式提供专门实现

### 示例
```cpp
template<typename T>
class A {};

template<>
class A<int> {};
```

## 9. auto 和 decltype 的区别？

### auto
根据初始化表达式推导变量类型。

### decltype
根据表达式规则推导类型，保留更多类型信息，常用于模板和泛型编程。

## 10. new 和 malloc 的区别？

### 标准回答
- `new`：分配内存并调用构造函数
- `delete`：调用析构函数并释放内存
- `malloc/free`：只负责原始内存分配与释放，不懂对象生命周期

### 面试加分点
`new` 表达式不等于 `operator new`：
- `new` 表达式 = 分配内存 + 构造对象
- `operator new` 只是底层分配接口

## 11. const 有哪些常见用法？

### 常见形式
- `const int* p`：指向常量的指针
- `int* const p`：常量指针
- `const int* const p`：常量指针，指向常量

### 成员函数后置 const
```cpp
int size() const;
```
表示该成员函数不修改对象的逻辑状态。

## 12. static 有哪些作用？

### 局部 static
函数内生命周期贯穿程序运行期，但作用域仍在函数内。

### 类内 static 成员
属于类而不属于具体对象实例。

### 文件级 static
限制符号链接可见性到当前翻译单元（传统 C/C++ 语义）。

## 13. inline 真正作用是什么？

### 标准回答
现代 C++ 中，`inline` 不只是“建议编译器内联”，更重要的是允许函数或变量在多个翻译单元中重复定义而不违反 ODR（单一定义规则）的特定条件。

### 易错点
不要把 inline 简单理解为“编译器一定把函数展开”。是否真正内联是编译器优化决策。

## 14. 什么是 ODR（单一定义规则）？

### 标准回答
ODR 要求某些实体在整个程序中只有一个定义，或多个定义必须完全一致，否则会导致链接错误或未定义行为。

### 高频场景
- 头文件中定义普通全局变量导致重定义
- 头文件中函数未加 `inline` 导致重复定义

## 15. 复习提示

这一章面试官很喜欢拿来区分“会写 C++”和“懂 C++ 工程化”：
- 容器不能只背复杂度
- 模板不是宏
- `new` 不等于 `malloc`
- `inline` 的意义不只是性能
